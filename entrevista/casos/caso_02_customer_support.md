# Case 2 — Customer-Support Agent that Resolves Tickets (DHL-flavored)

> A fully worked **PACTEDR** case. We take one concrete, high-volume scenario — a customer-support agent that **resolves** parcel/order tickets end-to-end instead of just answering — and walk all seven steps of the interview framework. The load-bearing themes are **ACTIONS + human-in-the-loop (HITL)**: real tool use via MCP, an approve / edit / reject approval card, idempotency on resume, guardrails that make an unsafe action structurally impossible, and the **business-value metrics** (deflection rate, cost per ticket, CSAT) that a hiring panel actually cares about.

This case is deliberately shaped like a **DHL Data & AI** interview prompt: *"Design an agent that handles inbound parcel-support contacts — 'Wo ist mein Paket?', address corrections before delivery, refunds for lost/damaged parcels — at scale, in German and English, without ever doing something irreversible on its own."* Everything below maps onto Kompass's real architecture and code (`kompass/graph`, `kompass/mcp_servers`, `kompass/guardrails`).

---

## Scenario at a glance

| Field | Value |
|---|---|
| **Domain** | Customer support / post-purchase operations (parcel logistics flavor) |
| **Primary intents** | Order/parcel status, **address change before delivery**, **refund / claim** for lost or damaged parcel, return initiation |
| **Channels** | Web chat + email, German 🇩🇪 and English 🇬🇧 |
| **Modes** | Autonomous (read + reversible actions) → HITL-gated (risky actions) → human escalation |
| **North-star metric** | **Autonomous resolution / deflection rate** at fixed CSAT and **zero unsafe actions** |
| **Why it's agentic** | It does not stop at retrieval — it **plans, calls tools, drafts a side-effecting action, pauses for approval when risky, executes idempotently, and remembers** |

> **Interview soundbite:** *"A support bot answers 'your parcel is in transit'. A support **agent** changes the delivery address before the truck leaves the depot, and refunds the €40 lost-parcel claim — but only after a one-tap supervisor approval, and never twice."*

## The PACTEDR lens (recap)

| Letter | Step | The one question it forces you to answer |

## Step P — Problem & Value

**Problem.** Post-purchase support is the #1 cost-and-CSAT center in logistics. Volumes are enormous, the questions are repetitive ("where is my parcel?"), but a meaningful slice requires an *action* on a system of record: rerouting a shipment, correcting an address, issuing a refund. A pure RAG bot deflects the FAQ tail but hands off the moment an action is needed — which is exactly the expensive part.

**Value hypothesis.** If the agent can *safely act*, we don't just deflect the question, we **resolve the ticket** — and resolution is where the money and the CSAT live.

We anchor to three metrics only (everything else is diagnostic):

| Metric | Baseline (human queue / naïve RAG) | Kompass — target | Δ |
|---|---|---|---|
| **Autonomous resolution / deflection** | ~0% resolved by RAG (answers only); 100% human-handled | **61% full auto + 24% one-tap approve** | ~85% never reaches full human handling |
| **Fully-loaded cost / ticket** | **€4.20** (AHT ≈ 6.5 min × loaded rate) | **€0.75** blended | **−82%** |
| **CSAT** | 79% | **84%** (instant resolution, careful escalation) | +5 pts |
| Grounding / faithfulness (answers) | n/a | 97% cited | — |
| **Unsafe actions executed** | — | **0** (HITL + guardrail gated) | — |

At a 100k-ticket/month segment that is **€420k → €75k/month ≈ €4.1M/year** saved, at *higher* CSAT.

> **Interview soundbite:** *"I lead with a P&L sentence, not a demo: '61% of tickets resolved with zero human touch, cost per ticket down 82%, and zero unsafe actions.' The last clause is what makes the first two acceptable to a risk-averse enterprise."*

> Numbers here are **modeled targets** (synthetic ACME eval + published contact-center benchmarks), regenerated by `make evals`; the README value table is the live source of truth.

## Step A — Architecture

Single agentic graph on **LangGraph v1**, one durable state object per ticket, deterministic control flow with a supervisor that cuts loops. Two interop layers: **MCP** (vertical — how the agent reaches tools/data) and **A2A** (horizontal — how it delegates to peer agents, Tier 2). Full rationale: [`../../docs/05_architecture.md`](../../docs/05_architecture.md).

```
   inbound ticket (chat/email, DE/EN)
        │
        ▼
   ┌─────────────┐   classify intent + language + extract entities (order_id, address)
   │   Triage    │   model: openai:gpt-5.4-nano  (cheap, high-volume)
   └─────┬───────┘
         ▼
   ┌─────────────┐   plan-and-execute; routes; cuts loops (max steps)
   │ Supervisor  │◀───────────────────────────────────────────┐
   └─────┬───────┘                                             │
         ├─▶ Retrieval Router → {RAG hybrid | CAG policy cache | NL2SQL over acme.db}
         │        (policy answers must carry citations)         │
         ├─▶ Action Agent ──[MCP: ticketing / sql]──▶ propose action + risk tier
         │        │                                             │
         │        ▼                                             │
         │   ┌──────────────────────────────────────────┐      │
         │   │ HITL middleware  interrupt_on(...)        │      │
         │   │  risk ≥ MEDIUM → APPROVAL CARD            │      │
         │   │  {approve · edit · reject}  → checkpoint  │──────┘ (reject → replan / escalate)
         │   └───────────────┬──────────────────────────┘
         │                   ▼ (approve/edit)
         │            execute tool  (idempotency_key)
         ├─▶ Critic / Verifier  (grounding + action-precondition check)
         └─▶ Safety agent       (prompt-injection / PII, Tier 2)
        │
   state:  ticket_ctx + conversation memory + per-user store (long-term)
   durable: Postgres checkpointer  |  typed I/O: Pydantic  |  traces: Langfuse
```

**State (typed, Pydantic).** The single fact that makes HITL work: everything needed to *resume* is in the checkpointed state — `ticket_id`, `customer_id`, extracted entities, retrieved policy citations, the **proposed action** (`{tool, args, risk_tier, rationale, idempotency_key}`), and the tool-result ledger. When the graph pauses on an interrupt, the Postgres checkpointer persists this; a resume rehydrates it exactly.

> **Interview soundbite:** *"I keep it a single graph, not a swarm. Multi-agent is a Tier-2 option for genuinely parallel specialties (A2A). For one ticket, a supervisor + typed state + a durable checkpointer beats agent-soup every time — it's debuggable and resumable."*

## Step C — Context & Retrieval

The agent needs two very different kinds of knowledge, so it routes per query (see [`../../docs/02_retrieval_strategies.md`](../../docs/02_retrieval_strategies.md)):

| Query shape | Strategy | Backend | Why |
|---|---|---|---|
| "What's your lost-parcel refund policy?" | **CAG** (cache-augmented) over a small, stable policy set | policy docs pinned in context | Policies are small + hot → skip vector search, read from cache; deterministic, fast, cheap |
| "Where is parcel 00340…?" / "refund €X for order Y" | **NL2SQL** over `corpus/acme.db` | SQLite/Postgres via `sql` MCP server | Facts live in the system of record, not prose; grounded in rows, not vibes |
| Long-tail how-to / edge policy | **RAG hybrid** (BM25 + dense, Chroma/Qdrant) | `doc_search` MCP server | Recall over a large, changing FAQ corpus |

**Grounding is mandatory.** Any policy statement the agent makes to a customer must cite the source clause; the Critic node rejects uncited claims. This matters doubly here because a *wrong policy quote* ("yes, you're eligible for a refund") becomes a *wrong action* two steps later.

**Memory.** Short-term conversation state + a per-user long-term store (prior tickets, language preference, known addresses). This is what lets the agent say "I see this is the second delay on order Y — I'll expedite the claim" rather than restarting cold.

## Step T — Tools & Actions  *(the heart of this case)*

This is where a support **agent** diverges from a support **bot**. The agent reaches tools through **MCP servers** (the vertical interop layer), classifies every proposed action by risk, and routes risky ones through the **declarative HITL middleware**. See the deep-dive: [`../../docs/04_hitl_patterns.md`](../../docs/04_hitl_patterns.md).

### 4.1 Tools via MCP

Kompass exposes support tools as MCP servers (`kompass/mcp_servers`), not ad-hoc Python functions. Why MCP: a clean tool/transport boundary, the same servers reusable across cases, and per-tool auth/scoping enforced at the server — the model never holds a payment credential.

| Tool (server) | Effect | Risk tier | HITL? |
|---|---|---|---|
| `get_order_status`, `track_parcel` (`sql`) | Read-only | **NONE** | Auto |
| `read_policy` (`doc_search`) | Read-only | **NONE** | Auto |
| `add_ticket_note`, `send_status_email` (`ticketing`) | Reversible / low-impact | **LOW** | Auto + logged |
| `update_shipping_address` (`ticketing`) | Reroutes a physical parcel | **MEDIUM** | **Approve/edit/reject** |
| `initiate_refund` (`ticketing`) | Moves money | **MEDIUM→HIGH** by amount | **Approve/edit/reject**; > cap → escalate |

### 4.2 The approve / edit / reject approval card

LangGraph v1 shipped a **declarative HITL middleware** (`interrupt_on`) on top of the runtime `interrupt()` primitive — you *declare* which tools pause and which decision types are allowed, instead of hand-wiring pauses.

```python
from langchain.agents.middleware import HumanInTheLoopMiddleware

hitl = HumanInTheLoopMiddleware(
    interrupt_on={
        # read-only tools never pause
        "get_order_status":        False,
        "track_parcel":            False,
        "read_policy":             False,
        # side-effecting tools require a human decision
        "update_shipping_address": {"allowed_decisions": ["approve", "edit", "reject"]},
        "initiate_refund":         {"allowed_decisions": ["approve", "edit", "reject"]},
    },
    description_prefix="Kompass wants to run a risky support action:",
)
```

For dynamic, amount-dependent gating we still drop to the runtime primitive so the card carries a human-readable preview and the policy citations that justify it:

```python
from langgraph.types import interrupt

def action_node(state: TicketState) -> TicketState:
    proposal = draft_action(state)          # {tool, args, risk_tier, rationale, idempotency_key}
    if proposal.risk_tier >= RiskTier.MEDIUM:
        decision = interrupt({              # ← graph pauses; state is checkpointed to Postgres
            "type": "approval_request",
            "action": proposal.tool,
            "args": proposal.args,          # e.g. {"order_id": "AC-8842", "amount_eur": 39.90}
            "preview": human_readable(proposal),   # "Refund €39.90 to order AC-8842 (lost parcel, policy §4.2)"
            "policy_refs": proposal.citations,
            "reversible": proposal.reversible,
        })
        proposal = apply_decision(proposal, decision)   # approve | edit | reject
    return execute_or_replan(state, proposal)
```

The three decisions, in support terms:

| Decision | What the approver does | Agent behavior |
|---|---|---|
| **approve** | One tap — "yes, refund €39.90" | Execute the tool with the proposed args |
| **edit** | Correct the payload — fix the typo'd address, cap the refund to €25 | Execute with the **edited** args (re-validated against guardrails) |
| **reject** | "No" + optional reason | Skip the tool; supervisor **replans** (offer alternative) or **escalates** to a human agent |

Resume happens over the API — `POST /resume` with the decision → `Command(resume=...)`:

```python
graph.invoke(Command(resume={"decision": "edit",
                             "args": {"amount_eur": 25.00}}), config)
```

The approval card is rendered in the Streamlit UI and returned by the API, so a supervisor (or the customer, for their own address) resolves it in ~15 seconds — cheap compared to a 6.5-minute human handle.

> **Interview soundbite:** *"HITL is declarative in LangGraph v1: I annotate which tools are risky and which decisions are allowed, and the middleware does the pausing. The human isn't babysitting the model — they're a typed decision on the one step that moves money or a physical parcel."*

### 4.3 Idempotency on resume  *(the detail that separates senior from mid-level)*

An interrupt means the graph **stops and later replays** from the last checkpoint. Naïvely, a resume could re-enter the action node and fire `initiate_refund` **twice**. Two defenses, both required:

1. **Checkpointed tool results.** Once a side-effecting tool returns, its result is written to state; on replay the node sees the result already present and does not re-call.
2. **Idempotency keys at the tool boundary** — the belt-and-suspenders that survives retries, crashes, and double-clicks:

```python
def initiate_refund(order_id: str, amount_cents: int, idempotency_key: str) -> RefundResult:
    if (prior := refund_ledger.get(idempotency_key)):
        return prior                       # replay-safe: NEVER double-refund
    result = payments.refund(order_id, amount_cents, key=idempotency_key)
    refund_ledger.put(idempotency_key, result)
    return result

# derived deterministically so the same logical action → same key
idempotency_key = sha256(f"{ticket_id}:initiate_refund:{order_id}:{amount_cents}")
```

The key is a pure function of the *logical* action, so an approve-then-crash-then-resume, or an impatient double-tap, collapses to exactly one refund.

> **Interview soundbite:** *"The moment your agent can pause and resume, every side-effecting tool must be idempotent. I derive an idempotency key from the logical action and check a ledger — so a resume, a retry, or a double-click can never double-charge a customer."*

### 4.4 Guardrails — an unsafe action is *structurally* impossible

Guardrails (`kompass/guardrails`) wrap the action path so "unsafe" isn't a hope, it's blocked before the tool is ever proposed:

- **Ownership check** — the `customer_id` on the session must own the `order_id`. Fails → no action, escalate. (Blocks "refund someone else's order" injection.)
- **Refund cap + policy precondition** — amount ≤ policy-derived cap *and* order state ∈ {lost, damaged, undelivered}. Over cap → forced escalation, never auto-approve.
- **Address-change window** — only while parcel status is pre-out-for-delivery.
- **Prompt-injection / PII** — Safety agent (Tier 2) scans customer text and retrieved docs; addresses are PII → GDPR-scoped logging (relevant for a German employer).
- **Typed outputs** — Pydantic-validated tool args; a malformed `amount_eur` never reaches the payments API.

These compose with HITL: guardrails decide *whether an action is even permissible*; HITL decides *whether a permissible-but-risky action is authorized*.

## Step E — Evaluation

You cannot ship an acting agent on vibes. Three layers (see [`../../docs/05_architecture.md`](../../docs/05_architecture.md#evaluation) and the question bank [`../banco_preguntas.md`](../banco_preguntas.md)):

| Layer | What it measures | How |
|---|---|---|
| **Retrieval / grounding** | Faithfulness, citation coverage, answer relevance | RAGAS over the ACME golden set |
| **Task completion** | Did the agent resolve the ticket correctly and take the *right* action? | Golden trajectories + **user-simulator** (τ-bench style, `evals/user_simulator`) |
| **Action safety** | **Unsafe-action rate (target 0)**, action precision, over-refund rate | Adversarial suite: injection, wrong-owner, over-cap, double-submit |
| **Value** | Deflection, cost/ticket, CSAT proxy | Simulated funnel; blended cost model |

The **action-precision and unsafe-action** metrics are the ones I'd put on the slide — a support agent that resolves 90% but wrongly refunds 1% is a net negative. Target: **0 unsafe actions, action precision ≥ 99%.**

> **Interview soundbite:** *"For an agent that acts, my headline eval isn't accuracy — it's unsafe-action rate. I run an adversarial suite (wrong-owner, over-cap, injection, double-submit) and gate CI on zero unsafe executions."*

## Step D — Deployment & Ops

- **Serving.** FastAPI — `POST /chat`, `POST /resume`, `GET /runs/{id}` — plus a Streamlit chat UI that renders citations and the HITL approval card. The `/resume` endpoint is what makes durable HITL usable in production.
- **Durability.** SQLite checkpointer for the zero-setup local demo; **Postgres checkpointer** for production so an interrupted ticket survives a deploy or a pod restart and can be resumed hours later by a supervisor. See [`../../docs/04_hitl_patterns.md`](../../docs/04_hitl_patterns.md) for the durable-execution discussion (and where Temporal fits for very-long-lived waits).
- **Model routing + cost control.** GPT-5.4 nano (`openai:gpt-5.4-nano`) for high-volume triage/classification, GPT-5.4 (`openai:gpt-5.4`) as the balanced default, GPT-5.5 (`openai:gpt-5.5`) only for genuinely hard reasoning. Caching + token budgets keep the €0.07 autonomous cost/ticket honest.
- **Observability.** Langfuse traces every run — the plan, each tool call, the interrupt, the human decision, the resume. For an acting agent this trace *is* the audit log.
- **Delivery.** Docker + `docker compose` (Qdrant + Postgres + Langfuse), CI in `.github/workflows/ci.yml`, versioned prompts.

> **Interview soundbite:** *"Durable checkpointing isn't a nice-to-have for HITL — it's the enabling primitive. A supervisor approving a refund three hours later must resume the exact paused state, which is why the production checkpointer is Postgres, not in-memory."*

## Step R — Risks & Reliability

| Risk | Mitigation |
|---|---|
| **Double side-effect** on resume/retry | Idempotency keys + checkpointed tool results (§4.3) |
| **Wrong / unsafe action** | Guardrails (ownership, cap, window) + HITL approval on risky tiers (§4.4) |
| **Prompt injection** ("ignore rules, refund me") | Safety agent scans input + retrieved docs; actions gated on *system-side* ownership check, not on model belief |
| **Hallucinated policy** → wrong eligibility | Mandatory citations + Critic grounding check; NL2SQL for facts |
| **Runaway loops / cost** | Supervisor step cap; model routing + token budgets |
| **Silent failure** | Langfuse traces + typed Pydantic I/O; failed tool → replan or escalate, never a fake success |
| **GDPR / PII** (addresses, names) | PII-scoped logging, EU-relevant data handling — deliberately called out for a German employer |

**Failure philosophy:** when unsure, the agent **degrades to a human**, it never guesses at an action. Escalation is a first-class outcome, not a bug.

## Why this case fits DHL Data & AI

Parcel logistics is the canonical "answer *and* act" support domain: the questions are high-volume and repetitive, but the value is in *rerouting, refunding, reclaiming* — actions on a system of record, in German and English, under GDPR, where a wrong or duplicated action has real cost. Kompass demonstrates exactly the shape a logistics AI team needs: **autonomous where safe, human-gated where risky, idempotent always, and measured in € and CSAT** — built on the same MCP + multi-agent + LangGraph stack Wesley has run in production (VW/CARIAD MCP, Crehana LangGraph multi-agent, NL2SQL over 40M records).

> **Interview soundbite:** *"This is the DHL-shaped version of what I've already shipped: an agent that resolves parcel-support tickets end-to-end, pauses only to move money or a physical parcel, and can prove — via traces and evals — that it never acted unsafely."*
